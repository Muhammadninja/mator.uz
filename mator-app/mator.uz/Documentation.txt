Official Technical Documentation: MATOR Ecosystem
Status: Architecture Design and MVP (June 2026)

Version: 2.0 (AI-Native with Model Context Protocol support)

Core Stack: Node.js (NestJS / Fastify), TypeScript, PostgreSQL (Managed), React Native + Expo, Telegram Bot API, Anthropic Claude API (Sonnet / Haiku), MCP SDK, Photoroom API.

1. System Overview and Concept
MATOR is an intelligent e-commerce ecosystem for the automotive spare parts market in Uzbekistan, built entirely around an AI-Native architecture. The system consists of a client mobile application featuring a "Digital Garage", a Telegram bot for suppliers, and an orchestration backend managed by AI agents via the Model Context Protocol (MCP).

Key Architectural Innovation
Instead of hardcoded REST API application logic, the backend exposes declarative MCP servers (tools, database access, external Yandex APIs). An AI agent powered by Claude models dynamically combines these tools in real-time to process both customer inquiries in the chat and chaotic, unstructured content coming from sellers at local physical markets (bazars).

2. Global Data Architecture (PostgreSQL)
The database schema is designed to decouple the master catalog of parts (ensuring uniqueness via factory GM part numbers) from the dynamic stock inventories provided by various points of sale.

SQL
-- 1. Profiles of sellers / physical stores
CREATE TABLE sellers (
    id SERIAL PRIMARY KEY,
    tg_id BIGINT UNIQUE NOT NULL,         -- Telegram ID for authorization and bot sessions
    phone VARCHAR(20) NOT NULL,
    store_name VARCHAR(100),              -- Store location/name (e.g., "Sergeli, Block 5, Shop 24")
    market_name VARCHAR(100),             -- Market cluster: "Sergeli", "Farhadsky", etc.
    address_comment TEXT,                 -- Local landmarks for the Yandex delivery courier
    location_lat DOUBLE PRECISION,        -- Exact geo-coordinates for vehicle dispatch
    location_lng DOUBLE PRECISION,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. Master catalog of unique auto parts
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    gm_number VARCHAR(50) UNIQUE,         -- Unique SKU (Factory GM part number)
    title VARCHAR(255) NOT NULL,          -- AI-standardized product name
    car_model VARCHAR(100),               -- Recognized car model (Cobalt, Gentra, BYD, etc.)
    image_url TEXT,                       -- URL to the processed photo (white background, PNG) in S3/Cloud Storage
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 3. Inventory stocks and pricing (Junction table)
CREATE TABLE stocks (
    id SERIAL PRIMARY KEY,
    seller_id INT REFERENCES sellers(id) ON DELETE CASCADE,
    product_id INT REFERENCES products(id) ON DELETE CASCADE,
    price_uzs DECIMAL(12, 2) NOT NULL,    -- Current price in national currency (UZS)
    quantity INT DEFAULT 1,               -- Available stock at the physical point of sale
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(seller_id, product_id)
);

-- 4. Customer orders and logistics tracking
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INT NOT NULL,
    status VARCHAR(50) NOT NULL,          -- 'paid', 'courier_searching', 'in_delivery', 'delivered'
    yandex_claim_id VARCHAR(100),         -- Ticket ID in the Yandex Delivery API system
    delivery_cost DECIMAL(10, 2),
    total_cost DECIMAL(12, 2),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
3. Supplier AI-Pipeline Specification (Telegram Bot)
The Telegram bot serves as the primary gateway for database ingestion. The entire ingestion pipeline is asynchronous and orchestrated by a lightweight Claude 3.5 Haiku agent.

[Seller sends Photo + Text] ──► [Node.js Backend] ──► [Task Queue (BullMQ)]
                                                             │
        ┌────────────────────────────────────────────────────┴────────────────────────────────────────────────────┐
        ▼ (Step 1: Graphics)                                                                                      ▼ (Step 2: Context)
[Photoroom API]                                                                                           [Claude 3.5 Haiku + MCP]
• Object isolation/cutout                                                                                  • GM-number extraction
• background_color: "ffffff"                                                                              • Car model detection
• Shadow/distortion removal                                                                               • Seller data validation
        │                                                                                                         │
        ▼                                                                                                         ▼
[Upload clean PNG to Cloud Storage] ──────────────────────────────────────────────────────────────────────► [Database Upsert (PostgreSQL)]
AI Workflow via MCP in the Bot
Graphics Layer: The backend forwards the raw photo to the Photoroom API, strips the background, generates a studio-quality PNG on a solid white background, and saves it to cloud storage.

Analytical Layer: The text provided by the seller is routed to the AI agent. Utilizing the mator-db-mcp-server, the agent checks: "Does a part with this GM number already exist?"

Conflict Resolution Logic: If the seller's text lacks a GM number, the AI agent uses the MCP server to query similar names and sends an interactive clarification prompt back to the seller:

"Aka (Brother), I recognized a shock absorber for Cobalt. However, there are two variations in our database. Do you have the old or the new model?"

4. MCP Infrastructure Specification (Model Context Protocol)
The backend spins up a set of isolated MCP servers using the official @modelcontextprotocol/sdk. This grants the AI model a secure, native interface to execute internal system functions.

4.1. mator-db-mcp-server (Data Access)
Provides the agent with tools for catalog searching and data aggregation:

search_products(query, car_model): Full-text search for automotive parts.

get_product_by_gm(gm_number): Retrieves exact technical specifications for a product card.

update_stock(seller_id, product_id, price, quantity): Updates the price and stock availability for a specific seller.

4.2. mator-logistics-mcp-server (Yandex Delivery API Integration)
Manages express delivery workflows:

calculate_delivery_cost(from_lat, from_lng, to_lat, to_lng, cargo_type): Requests real-time delivery cost estimations from Yandex Delivery.

create_delivery_claim(order_id, seller_point, customer_point): Submits an official claim (POST /v1/claims/create) and dispatches a courier.

get_delivery_status(claim_id): Streams courier status updates (searching, arrived at store, en route to customer).

5. Client Application Architecture (React Native / Expo)
The user interface centers around a custom floating tab bar (4 tabs) and a robust Chat-UX that serves as an alternative to traditional catalog navigation.

Floating Tab Bar Configuration
Main (Home):

Search Bar + Product Categories (Engine, Suspension, Body, Electrical).

AI-Smart-Chat: An end-to-end conversational interface with Claude. The user describes their issue in natural language ➔ The agent calls mator-db-mcp-server and mator-logistics-mcp-server ➔ The agent compiles a tailored offer ➔ An interactive product widget with a purchase button renders directly inside the chat window.

My Garage:

An interface to add a vehicle using its license plate or VIN. This automatically applies a global filter across the marketplace catalog specifically for that vehicle.

Maintenance Tracker: Interactive wear-and-tear meters for consumables (oil, brake pads, spark plugs) hooked up to smart push-notification triggers indicating when a replacement is due.

Cart / Orders:

An aggregator for selected parts supporting local BNPL / installment plans (Uzum Nasiya, Anor, ZoodPay).

Realtime Map Tracking: A screen with an embedded map where courier coordinates are streamed via Yandex webhooks, allowing users to watch the delivery move from the seller directly to their location or service station.

Profile:

Personal info, maintenance history (digital service logbook), and a language toggle (Uzbek / Russian).
